🌲 Blocking Tree
SQL: Lock blocking tree
SELECT blocked.pid AS blocked_pid, blocked.usename AS blocked_user, blocking.pid AS blocking_pid, blocking.usename AS blocking_user, now() - blocked.query_start AS blocked_for, blocked_activity.query AS blocked_query, blocking_activity.query AS blocking_query FROM pg_catalog.pg_locks AS blocked_locks JOIN pg_catalog.pg_stat_activity AS blocked_activity ON blocked_activity.pid = blocked_locks.pid JOIN pg_catalog.pg_locks AS blocking_locks ON blocking_locks.locktype = blocked_locks.locktype AND blocking_locks.relation IS NOT DISTINCT FROM blocked_locks.relation AND blocking_locks.granted = true AND blocked_locks.granted = false AND blocking_locks.pid != blocked_locks.pid JOIN pg_catalog.pg_stat_activity AS blocking_activity ON blocking_activity.pid = blocking_locks.pid;

⚠ RUNBOOK: Emergency Failover
🚨 Use when primary pod is down or unresponsive. Patroni will auto-promote pg-standby-0 if TTL (30s) expires, but manual failover is faster.
Step 1 — Verify primary is down
patronictl status oc get pods -n postgres oc describe pod pg-primary-0 -n postgres
Step 2 — Force failover
patronictl failover prod-pgcluster-uae-ha \ --master pg-primary-0 \ --candidate pg-standby-0 \ --force
Step 3 — Verify new primary
patronictl status -- New leader should be pg-standby-0 TL:9
Step 4 — Verify PgBouncer reconnected
psql -h 127.0.0.1 -p 5432 -U pgbouncer pgbouncer SHOW POOLS;
Step 5 — Reinit old primary when recovered
patronictl reinit prod-pgcluster-uae-ha pg-primary-0
Step 6 — Confirm DR re-sync
patronictl status -- pg-dr-0 and pg-dr-1 should resume streaming TL:9 -- Both DC2 nodes reinit if needed: patronictl reinit prod-pgcluster-uae-ha pg-dr-1
⚠ RTO ~2-5 min. RPO ~0 (sync standby). Notify app team and change manager on call.

ℹ For planned maintenance. Zero-data-loss with sync replication. Expected downtime <30s for client reconnect.

patronictl switchover prod-pgcluster-uae-ha \ --master pg-primary-0 \ --candidate pg-standby-0 \ --scheduled now --force

When to use

Key steps

Identify target time from audit log SELECT ts, command_tag, object_name FROM pgaudit.log WHERE command_tag IN ('DROP','TRUNCATE','DELETE') AND ts > now() - interval '24h' ORDER BY ts DESC; -- 2. Verify data SELECT count(*) FROM dropped_table; -- 5. Export and re-import to production pg_dump -t dropped_table recovered_db > recovered.sql psql hplus_core < recovered.sql -- 6. 🔑 Role & Privilege Audit SQL
List all roles and attributes
SELECT rolname, rolsuper, rolcreaterole, rolcreatedb, rolcanlogin, rolreplication, rolconnlimit, rolvaliduntil FROM pg_roles ORDER BY rolname;
Table-level privileges
SELECT grantee, table_schema, table_name, string_agg(privilege_type,', ') AS privileges FROM information_schema.role_table_grants WHERE grantee NOT IN ('postgres','PUBLIC') GROUP BY 1,2,3 ORDER BY 1,2,3; -- Column-level privileges SELECT grantee, table_name, column_name, privilege_type FROM information_schema.column_privileges WHERE grantee NOT IN ('postgres','PUBLIC') ORDER BY 1,2,3; -- Check if role can connect to specific DB SELECT has_database_privilege('hplus_app','hplus_core','CONNECT'); -- Role membership tree SELECT r.rolname, m.rolname AS member_of FROM pg_auth_members am JOIN pg_roles r ON r.oid = am.roleid JOIN pg_roles m ON m.oid = am.member ORDER BY 2,1; -- Password expiry audit SELECT rolname, rolvaliduntil, CASE WHEN rolvaliduntil < now() THEN 'EXPIRED' WHEN rolvaliduntil < now()+interval '30d' THEN 'EXPIRING SOON' ELSE 'OK' END AS status FROM pg_roles WHERE rolcanlogin ORDER BY rolvaliduntil;
